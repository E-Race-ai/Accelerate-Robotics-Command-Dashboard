Git & GitHub

Git is the system that tracks every change to our code. GitHub is the website where that history lives so the team can collaborate.

Repositoryaka "repo"

The folder of code for one project, plus its full change history. We have several — accelerate-robotics, accelerate-elevator, accelerate-thesis-hotel, accelerate-carts.

Think of it like: a Google Doc that remembers every edit anyone has ever made, by anyone, forever.
Clone

Download a full copy of a repo to your laptop so you can read or change the code locally.

Why we say it: "I cloned the repo" = "I have the code on my machine now."
Fork

Make your own copy of someone else's repo on GitHub. Used mostly for open-source. Internally we work on branches in the same repo, not forks.

Branch

A named lane of changes running parallel to the main code. You start a branch, make changes there, then merge it back. Multiple people can work on different branches at once without stepping on each other.

Think of it like: a draft that doesn't affect the published version until everyone agrees it's good.
main

The single branch that represents what's actually live. Production deploys come from here. We never commit directly to main — every change first lives on its own branch and gets reviewed before it merges in.

Fresh branch

A brand-new branch started from the latest main — a clean slate for one specific piece of work. Starting "fresh" matters because if your branch is based on stale code, your changes could conflict with what's already shipped.

Commit

A single saved snapshot of changes, with a short message explaining what changed and why. The Activity Calendar on the Command Center counts commits.

Think of it like: hitting "Save" with a sticky note attached.
Push

Send your local commits up to GitHub so others can see them and so they're backed up off your laptop.

Pull

Download new commits from GitHub onto your laptop, so your local copy is up to date with what teammates have shipped.

Pull requestaka "PR"

A formal proposal on GitHub that says "please merge my branch into main." It shows the diff, lets reviewers comment line by line, and runs the test suite. Once it's approved and tests pass, it merges.

Why "pull": you're requesting that main pull your changes in.
Merge

Combine the changes from one branch into another. The most common merge is "feature branch → main" at the end of a PR.

Squash

When merging a PR, collapse all of its commits into one tidy commit on main. Keeps the history readable. We squash-merge by default.

Rebase

Replay your branch's commits on top of the latest main, so it's "current" before merging. Useful when main has moved while you were working.

Conflict"merge conflict"

When two branches changed the same line of the same file and Git can't decide which wins. A human resolves it by picking which version to keep (or blending them).

Stash

Temporarily set aside uncommitted changes so you can switch branches, then bring them back later. Like a pocket for unfinished work.

Diff

The line-by-line view of what changed — red lines were removed, green lines were added. PRs and code reviews are mostly about reading diffs.

HEAD

Git's word for "where you are right now" — the most recent commit on the branch you're currently on.

Origin

The default name for the GitHub copy of the repo your laptop talks to when you push or pull.

Tag / release

A stable label stuck on a specific commit, usually a version number like v1.4.0. Used for cutting versions and for rollbacks.

Code review & deploys

How code gets from a teammate's laptop to the live site.

Code review

Another teammate reads your PR's diff and either approves it or asks for changes. Required before anything merges to main.

CIContinuous Integration

The automated test/lint runner that fires on every PR. Green checkmark = safe to merge. Red X = something broke; fix before merging.

CDContinuous Deployment

Auto-deploy to production whenever main advances. No manual "push to prod" step.

Deploy

Push the new code out to where users can see it. Usually triggered automatically by a merge to main.

Production"prod"

The live site real users hit. Most cautious changes go through staging first.

Staging

A near-identical copy of production used for last-mile testing — same code paths, but no real users yet.

Dev / development

Your laptop — where work-in-progress lives before going to staging or prod.

Hotfix

An urgent fix that goes straight to main and prod, skipping the usual rhythm because something is on fire.

Rollback

Re-deploy the previous version because the current one is broken. The fastest way out of a bad release.

Lint

An auto-checker that flags risky or stylistically inconsistent code patterns. Runs in CI alongside tests.

Test (unit, integration, E2E)

Unit tests one function in isolation. Integration tests a few pieces working together (e.g. an API endpoint hitting the database). End-to-End (E2E) tests the whole app like a real user — clicking buttons in a real browser.

Build

The step that bundles, compiles, or minifies code for shipping. We have a minimal build — most of our code ships as written.

Auto-merge

Tell GitHub: "merge this PR automatically the moment CI passes and reviewers approve." Saves babysitting.

Commit message types

Every commit message starts with a type so you can scan history at a glance.

feat

A new capability for users — a feature being added.

fix

A bug fix.

refactor

Internal restructure with no behavior change. Same outcome, cleaner code.

docs

Documentation only — no code change.

test

Adding or fixing tests.

chore

Housekeeping — dependency bumps, config tweaks, file renames.

Software & data

Pieces of the app you'll hear named.

API

The set of URLs the front-end (and other tools) call to read or change data on the server. Each URL is an "endpoint."

Endpoint

One specific URL on the server, like /api/deals. Each endpoint does one thing — list deals, create a deal, etc.

Frontend

The part that runs in your browser — what users see and click.

Backend

The part that runs on a server. It stores data, sends emails, and serves the frontend.

Database

Where data is stored permanently. We use SQLite — a single file on disk that the backend reads from and writes to.

Schema

The shape of the database — which tables exist, what columns each has, and which fields are required.

Migration

A script that updates the database schema (adds a column, renames a table, etc.) without losing existing data.

Cache

Saved-aside data so we don't have to recompute or re-fetch it. Faster, but can go stale — gets refreshed on a timer.

JWTJSON Web Token

The signed token that proves you're logged in as an admin. Lives in a secure browser cookie, expires after 24 hours.

Edge case

A rare or unusual scenario that often breaks code if it isn't explicitly handled (empty inputs, two users editing at once, etc.).

Robots & elevators

Selected terms — full list in docs/00-overview/glossary.md.

AMR

Autonomous Mobile Robot — any wheeled floor-going robot that localizes and navigates on its own.

Button emulator"the wedge"

Our universal elevator-integration product — a small board that wires in parallel to existing push-buttons and lets a robot "press" them over BLE/LoRa, without touching the elevator's firmware.

E-Box

Keenon's proprietary robot-elevator bridge hardware. Master + slave + RFID tags. Vendor-locked.

LiDAR

Light Detection and Ranging — a laser scanner robots use to map their surroundings and avoid obstacles.

LoRa

Long Range, low-power radio in the 850–930 MHz band. Penetrates elevator shafts where Bluetooth can't.

SLAM

Simultaneous Localization and Mapping — how robots build internal floor plans while figuring out where they are inside them.

Hall call / Car call

Hall call = pressing the up/down button at the elevator lobby. Car call = pressing a floor number from inside the cab.

RaaS

Robot-as-a-Service — leasing robots bundled with management and support, instead of selling them outright.

Business

The acronyms you'll see in pipeline notes and proposals.

ARR / MRR

Annual / Monthly Recurring Revenue. The portion of revenue that repeats predictably (subscriptions, RaaS).

Capex / Opex

Capex = capital expenditure (one-time hardware buys, build-outs). Opex = ongoing operational expenditure (service, maintenance).

KOL

Key Opinion Leader — a respected expert whose endorsement drives industry adoption.

LOI

Letter of Intent — a non-binding signal that someone plans to do business with us.

POC

Proof of Concept — a small deployment to demonstrate the idea works before committing to the full project.

SLA

Service Level Agreement — the contract clause specifying how fast we respond and how often the system is up.

TAM

Total Addressable Market — the maximum revenue if we captured 100% of every potential customer.
