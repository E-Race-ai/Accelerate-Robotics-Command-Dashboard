The Problem

Controller-Level Integration Is a Bottleneck

Today, getting a robot to use an elevator requires expensive, proprietary API integration with the elevator manufacturer. This creates a deployment bottleneck that delays hospital robotics programs by months.

Controller-Level Integration

  • $5,000–$15,000 per elevator
  • Requires elevator OEM partnership (Otis, KONE, Schindler, TK)
  • Proprietary protocols — locks you to one elevator brand
  • Weeks-to-months of IT/security approval
  • Modifies elevator controller firmware or wiring
  • May require full recertification (ASME A17.1)
  • Elevator downtime during installation

Button-Level Emulation

  • ~$23 per floor in hardware
  • Works with ANY elevator brand — universal
  • Simple dry contact relay — mimics a finger press
  • 30-minute install by elevator technician
  • Zero changes to elevator controller or logic
  • No recertification — equivalent to adding an access control reader
  • Human buttons continue to work normally

System Architecture

How the Emulator Works

The emulator installs in parallel with the existing buttons. It receives wireless commands from the robot via BLE and closes a solid-state relay to simulate a button press. The elevator controller sees an identical signal to a human finger press.

Hardware Approaches

Four Paths to a Working Prototype

From quick prototype to production-ready module — each approach has different tradeoffs in cost, time, and scalability.

01

ESP32-C3 + Solid-State Relay

Smallest, most reliable, production-ready. Two MOSFET photo relays provide silent, wear-free switching with 1500V isolation.
  • MCU Seeed XIAO ESP32C3 (21×17.5mm)
  • Relay 2× OMRON G3VM-61GR2 (SOP-4, 3.9mm)
  • Power Pololu D24V5F3 (24V→3.3V, 13×10mm)
  • Wireless WiFi 802.11n + BLE 5.0
  • Draw ~4mA idle / ~27mA peak (at 24V input)
  • Size ~40×25mm total footprint
  • Cycle life Unlimited (no moving parts)
~$23 per floor
02

ESP32 + Mechanical Relay Board

Faster to prototype with off-the-shelf 2-channel relay module. Audible click, limited cycle life, larger footprint.
  • MCU ESP32-C3 Super Mini ($3)
  • Relay 2-ch 3.3V relay board (SRD-03VDC)
  • Power MP1584EN buck module (28V max)
  • Wireless WiFi + BLE 5.0
  • Draw ~120mA per relay + MCU
  • Size ~70×17mm relay + MCU
  • Cycle life ~100,000 operations
~$15 per floor
03

Flipper Zero + Relay Hat

Use Flipper Zero's GPIO with a transistor driver board to trigger relays. Overkill but useful as a development/testing tool.
  • Device Flipper Zero ($169)
  • GPIO 8 pins, 3.3V, 20mA max
  • Relay Via NPN transistor (2N2222) driver
  • Wireless BLE, Sub-GHz, NFC, IR
  • Limitation Battery-powered, no WiFi
  • Best for Signal analysis & BLE testing
  • Production Not suitable — dev tool only
~$185 per unit
04

Custom PCB (Scale Production)

Single PCB with ESP32-C3-MINI-1 module, two G3VM-61GR2 relays, and integrated buck converter. Target: <$10/unit at volume.
  • MCU ESP32-C3-MINI-1 (13.2×16.6mm)
  • Relay 2× G3VM-61GR2 (on-board)
  • Power TPS62743 or similar (on-board)
  • Target size 30×20mm single PCB
  • Target cost <$10 BOM at 1,000 units
  • Extras Polyfuse, input cap, antenna
  • Timeline 4–6 weeks for PCB fab + assembly
<$10 at scale

Build Guide

Step-by-Step: From Unboxing to Working Prototype

Follow these steps to build a working elevator button emulator on your bench. No prior electronics experience required — just follow the wiring and flash the code.

Step 1 — Arduino IDE Setup

Install Software & Configure Board

The XIAO ESP32C3 is programmed via Arduino IDE. You need to add the ESP32 board package first.

  1. Download and install Arduino IDE 2.x (free, Mac/Windows/Linux)
  2. Open Arduino IDE → File → Preferences
  3. In "Additional Boards Manager URLs", paste this URL:
    https://raw.githubusercontent.com/espressif/arduino-esp32/gh-pages/package_esp32_index.json
  4. Go to Tools → Board → Boards Manager → search "esp32" → install the Espressif ESP32 package (latest version)
  5. Go to Tools → Board → ESP32 Arduino → scroll down and select "XIAO_ESP32C3"
  6. Connect the XIAO via USB-C cable. Go to Tools → Port → select the new serial port that appears
Tip: If the port doesn't appear, hold the BOOT button on the XIAO while plugging in the USB cable, then release. This forces bootloader mode. After uploading, press RESET.

Reference: Seeed XIAO ESP32C3 Getting Started Guide

Step 2 — Wiring on Breadboard

Connect All Components

Wire everything on a breadboard first. The circuit has three sections: power, relay 1 (UP), and relay 2 (DOWN).

XIAO ESP32C3 Pin Assignments

XIAO PinGPIOConnected ToPurpose
3V3—Pololu VOUT3.3V power input from buck converter
GND—Pololu GND + SSR pin 2 (both)Common ground
D1GPIO3100Ω resistor → SSR1 pin 1 (anode)UP relay trigger
D10GPIO10100Ω resistor → SSR2 pin 1 (anode)DOWN relay trigger
Why D1 and D10? These GPIOs (3 and 10) have no boot-strapping conflicts. Avoid D0 (GPIO2), D8 (GPIO8), and D9 (GPIO9) — they are strapping pins that can interfere with boot if pulled low.

Step 3 — Flash the Firmware

Upload BLE GATT Server Code

This firmware makes the ESP32 advertise as a BLE device. When a robot (or your phone) connects and writes a command, it pulses the appropriate relay for 200ms.

// ============================================================ // Elevator Button Emulator — BLE GATT Server // Target: Seeed XIAO ESP32C3 // Drives 2x OMRON G3VM-61GR2 SSRs via GPIO3 (UP) and GPIO10 (DN) // ============================================================ #include <BLEDevice.h> #include <BLEUtils.h> #include <BLEServer.h> // UUIDs — generate unique ones per floor for production #define SERVICE_UUID "e1e10001-0001-4b1a-b000-acceler8lift" #define CHAR_CMD_UUID "e1e10001-0002-4b1a-b000-acceler8lift" #define CHAR_STATUS_UUID "e1e10001-0003-4b1a-b000-acceler8lift" // GPIO pins — D1 (GPIO3) and D10 (GPIO10) are safe, no boot conflicts #define PIN_UP 3 // XIAO D1 → SSR1 → elevator I/O#1 (UP) #define PIN_DOWN 10 // XIAO D10 → SSR2 → elevator I/O#2 (DOWN) // How long to hold the relay closed (simulates button press duration) #define PULSE_MS 200 // 200ms — typical elevator debounce accepts 100-500ms BLECharacteristic *pCmdChar; BLECharacteristic *pStatusChar; bool deviceConnected = false; // Connection callbacks — track when robot connects/disconnects class ServerCallbacks : public BLEServerCallbacks { void onConnect(BLEServer* pServer) { deviceConnected = true; Serial.println("[BLE] Device connected"); } void onDisconnect(BLEServer* pServer) { deviceConnected = false; Serial.println("[BLE] Device disconnected — restart advertising"); BLEDevice::startAdvertising(); } }; // Command write callback — this fires when robot sends UP/DOWN command class CmdCallbacks : public BLECharacteristicCallbacks { void onWrite(BLECharacteristic *pCharacteristic) { std::string value = pCharacteristic->getValue(); if (value.length() > 0) { uint8_t cmd = value[0]; if (cmd == 0x01) { // UP command — pulse SSR1 Serial.println("[CMD] UP — pulsing I/O#1"); digitalWrite(PIN_UP, HIGH); delay(PULSE_MS); digitalWrite(PIN_UP, LOW); pStatusChar->setValue("UP_OK"); pStatusChar->notify(); } else if (cmd == 0x02) { // DOWN command — pulse SSR2 Serial.println("[CMD] DOWN — pulsing I/O#2"); digitalWrite(PIN_DOWN, HIGH); delay(PULSE_MS); digitalWrite(PIN_DOWN, LOW); pStatusChar->setValue("DN_OK"); pStatusChar->notify(); } else { Serial.printf("[CMD] Unknown: 0x%02X\n", cmd); } } } }; void setup() { Serial.begin(115200); Serial.println("Elevator Button Emulator v1.0"); // Configure relay GPIO pins as outputs, start LOW (relays off) pinMode(PIN_UP, OUTPUT); pinMode(PIN_DOWN, OUTPUT); digitalWrite(PIN_UP, LOW); digitalWrite(PIN_DOWN, LOW); // Initialize BLE BLEDevice::init("ELEV-F01"); // Change per floor: ELEV-F01, F02, etc. BLEServer *pServer = BLEDevice::createServer(); pServer->setCallbacks(new ServerCallbacks()); BLEService *pService = pServer->createService(SERVICE_UUID); // Command characteristic — robot writes 0x01 (UP) or 0x02 (DOWN) pCmdChar = pService->createCharacteristic( CHAR_CMD_UUID, BLECharacteristic::PROPERTY_WRITE ); pCmdChar->setCallbacks(new CmdCallbacks()); // Status characteristic — emulator notifies robot of result pStatusChar = pService->createCharacteristic( CHAR_STATUS_UUID, BLECharacteristic::PROPERTY_READ | BLECharacteristic::PROPERTY_NOTIFY ); pStatusChar->setValue("READY"); pService->start(); // Start advertising — robot will scan for this BLEAdvertising *pAdvertising = BLEDevice::getAdvertising(); pAdvertising->addServiceUUID(SERVICE_UUID); pAdvertising->setScanResponse(true); pAdvertising->setMinPreferred(0x06); // Helps with iPhone connection BLEDevice::startAdvertising(); Serial.println("[BLE] Advertising as ELEV-F01 — waiting for connections..."); } void loop() { // BLE callbacks handle everything — nothing needed in loop delay(100); }
  1. Open Arduino IDE → File → New Sketch
  2. Delete all existing code and paste the code above
  3. Make sure Tools → Board is set to "XIAO_ESP32C3" and the correct Port is selected
  4. Click the Upload button (→ arrow). Wait for "Done uploading."
  5. Open Tools → Serial Monitor (set baud rate to 115200). You should see:

Step 4 — Test with Your Phone

Use nRF Connect to Simulate Robot Commands

Before connecting to the elevator, verify the BLE service works using the free nRF Connect app on your iPhone.

  1. Install nRF Connect from the App Store (free, by Nordic Semiconductor)
  2. Open the app → tap SCAN. Look for "ELEV-F01" in the device list
  3. Tap CONNECT next to ELEV-F01. The app will discover the GATT services.
  4. Expand the service with UUID e1e10001-0001-...
  5. Find the Command characteristic (UUID ending in 0002). Tap the up-arrow icon (Write)
  6. In the write dialog:
    • Select format: UINT8 or Hex
    • Enter 01 and tap SEND → UP relay should pulse (check Serial Monitor)
    • Enter 02 and tap SEND → DOWN relay should pulse
  7. Watch the Serial Monitor — you should see [CMD] UP — pulsing I/O#1 or [CMD] DOWN — pulsing I/O#2
LED Test: For initial testing without the elevator, temporarily connect an LED + 330Ω resistor between each SSR output (pins 3-4). When you send the BLE command, the LED will flash for 200ms — confirming the relay is switching.
Step 5 — Connect to Elevator Signal Fixture

Wire the Emulator to the ThyssenKrupp Screw Terminal

You already have the disassembled fixture. Connect the prototype in parallel with the existing buttons.

Safety First: When working with a live elevator fixture, have a licensed elevator technician present. The 24V circuit is low-voltage and safe to touch, but proper procedure protects you legally and ensures code compliance.

Screw Terminal Connections

Screw TerminalLabelConnect ToWire
Terminal 1 I/O#2 (DOWN) SSR2 Pin 3 (output A) Blue
Terminal 2 I/O#1 (UP) SSR1 Pin 3 (output A) Yellow
Terminal 3 G24 (Ground) SSR1 Pin 4 + SSR2 Pin 4 + Pololu GND + XIAO GND Black
Terminal 4 P24 (+24V) Polyfuse input → Pololu VIN Red
  1. Power off is NOT required — the emulator connects in parallel. But for your first connection, disconnect the USB from the XIAO.
  2. Loosen the screw terminals on the fixture and insert your wires alongside the existing wires (do NOT remove the existing wires)
  3. Red wire from P24 → through polyfuse → to Pololu VIN. Black wire from G24 → to Pololu GND.
  4. The Pololu's 3.3V output powers the XIAO through its 3V3 pin. Remove the USB cable — the XIAO now runs from elevator power.
  5. SSR1 output (pins 3+4) bridges I/O#1 to G24. SSR2 output bridges I/O#2 to G24.
  6. Test: Open nRF Connect → connect to ELEV-F01 → send 01. The elevator UP button LED on the faceplate should illuminate for 200ms.
  7. Press the physical UP button by hand — it should still work normally (parallel connection preserves human operation).
Step 6 — Validate & Measure

Confirm Everything Works Before Deployment

  1. BLE range test: Walk away from the fixture with nRF Connect open. Verify you can still trigger the relay at 10+ meters. Test through a wall/door.
  2. Current measurement: If you have a multimeter, measure the current draw on the P24 line. Expected: ~4mA idle, ~24mA during BLE TX bursts.
  3. Relay timing: The 200ms pulse should be long enough for the elevator controller to register. If not, increase PULSE_MS to 300 or 500 in the firmware.
  4. Human button coexistence: While the emulator is connected, press the physical buttons repeatedly. Verify they still work normally and the emulator doesn't interfere.
  5. Power cycle test: Disconnect and reconnect the 24V supply. The ESP32 should reboot and start advertising within 2 seconds.
  6. Multiple triggers: Send 10 rapid UP commands from nRF Connect. All should execute without the ESP32 crashing or becoming unresponsive.
Step 7 — Add OTA Updates (Optional)

Over-the-Air Firmware Updates via BLE

Once installed in an elevator fixture, you won't want to open the panel every time you update firmware. BLE OTA lets you push new code from your phone or laptop.

  1. In Arduino IDE: Tools → Manage Libraries → search "BLE OTA Update" (by RaghavG1999) → Install
  2. Add OTA capability to the firmware — see GitHub: bluetooth_ota_firmware_update
  3. Alternative: fbiego/ESP32_BLE_OTA_Arduino — faster uploads (~12 KB/s), has companion Android app
  4. For web-based OTA: SparkFun: ESP32 OTA over BLE via Web Bluetooth
Step 8 — Package for Installation

Solder, Enclose, and Label

  1. Transfer from breadboard to perfboard: Solder all components to a small perfboard. Keep the layout compact — target under 40×25mm.
  2. Mount in enclosure: Drill holes for the 4 signal wires (P24, G24, I/O#1, I/O#2) and optional USB-C access hole for debugging.
  3. Label the box: Mark with floor number, BLE name (ELEV-F01), and wire color legend.
  4. Strain relief: Use a cable gland or zip-tie the wires where they exit the enclosure to prevent disconnection from vibration.
  5. Install inside fixture: The module fits in the cavity behind the ThyssenKrupp faceplate, alongside the existing SMC modules. Secure with double-sided tape or a small bracket.

Safety & Compliance

Why This Is Inherently Safe

The emulator can only do what a human finger can do — press UP or DOWN. It cannot override, cancel, or interfere with any safety system. A spurious hall call is a nuisance, not a hazard.

🛡

Zero Controller Modification

The elevator's controller firmware, logic, and wiring remain completely untouched. The emulator operates at the signaling layer only — identical to a parallel button or access control reader.

🚨

Fire Recall Unaffected

ASME A17.1 Phase I Emergency Recall operates at the controller level and overrides ALL button inputs. The emulator's relay closures are ignored during fire recall — exactly like physical buttons.

⚡

Galvanic Isolation

OMRON G3VM-61GR2 provides 1500Vrms optical isolation between the ESP32 circuit and the elevator signaling circuit. No electrical path between the microcontroller and elevator wiring.

👋

Human Buttons Still Work

Parallel connection means the physical buttons continue to function normally. If the emulator loses power, fails, or is removed, human operation is completely unaffected.

📋

ASME A17.1 Classification

Adding a parallel contact to a hall call signaling circuit is analogous to installing an access control card reader — a common retrofit classified as a "minor alteration" requiring elevator tech supervision and AHJ inspection.

🏥

Hospital Code Blue Compatible

Code Blue elevator override operates at the controller level. The emulator's hall call signals are overridden just like physical button presses during emergency modes.
