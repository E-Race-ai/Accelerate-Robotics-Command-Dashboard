Pair your B10

Follow these steps — the whole thing takes about 15 seconds.
1
Press the B10 button for ~1 second
You'll feel a short vibration buzz and see the blue LED blink twice. That's discoverable mode.
2
Within 5 sec, click "Find my B10"
Match the MAC. Flip your B10 over — the 12-character ID printed on the back (e.g. AC233FABF61A) is your B10's MAC address. In Chrome's picker, that MAC appears in parentheses next to "Unknown or Unsupported Device" — formatted as AC:23:3F:AB:F6:1A. Pick the entry whose MAC matches. If you can't tell which is yours, try each — we now auto-detect non-B10 picks and tell you to retry without disconnecting the page.
3
Pick it and hit Pair
We auto-validate it's a real Minew B10 by reading the manufacturer field. If you picked the wrong one, you'll get a red banner and the page disconnects — click Connect B10 again and pick a different entry. Wrong picks are remembered so you don't retry them.
Can't identify it in the list? The B10 commonly shows up with no name (just a MAC). Best strategies:
  1. Press the button again right before clicking Find — the 60s window starts from the press, not from when you open this modal.
  2. Watch for the device appearing live — the Chrome picker scans continuously. Press the B10 while the picker is open and watch for a new entry to pop in.
  3. Try every "Unknown or Unsupported Device" — one of them is almost certainly your B10. We can disconnect and retry if you pick wrong.
  4. If nothing ever appears: toggle macOS Bluetooth off/on, or plug in the charger for 30 sec to wake a flat battery.

How this fits Accelerate Robotics

The B10 is a physical trigger primitive: a cheap, rugged, IP66, 60-day battery wearable with 3 distinct button gestures + motion. In the hospital robotics OS, it's one of the input modalities that triggers fleet action:

  • Guest / patient wearable → personal request or alarm that routes to staff + robots
  • Staff wearable → duress, assist, or workflow trigger
  • Robot / cart tag → presence and proximity beacon for indoor positioning
  • Event primitive → any physical moment we want to become a software event

Algorithm provenance & upgrade path

Cole-Kripke is accelerometer-only and gives wake/sleep — useful but coarse. To get staged wake/NREM/REM output you need a second signal (HR or HRV). Upgrade options that pair cleanly with B10 + a chest-strap HR sensor: Walch et al. Apple Watch model (open, accel + HR → wake/NREM/REM), SleepECG (BSD-3, single-lead ECG), and SleepKit (BSD-3, designed for low-power MCUs — direct B10 fit). For PSG-grade staging in a Python sidecar: YASA (BSD-3, 87% PSG agreement) or U-Sleep (MIT, channel-agnostic).

Algorithm provenance

This is a 1:1 port of AtlasTurnMonitorWatch/Services/MotionService.swift's calculateRawRoll() + BodyPosition.classify() from the Atlas iOS codebase. Roll = atan2(g.x, g.z) * 180/π (rotation around the wrist's y-axis = patient body roll). Six clinical classes: supine (|r|<15°), slight L/R (15–45°), lateral L/R (45–155°), prone (>155°). Gravity is low-pass filtered (α=0.2) to suppress sensor noise. The Q2h threshold is the standard repositioning interval cited in the Vollman/Black HAPI prevention literature.

When this card is wired to a worn B10 (or an Atlas Patch via the same accel pipeline), every position change emits an Open mHealth datapoint that BMM/UMM/Mobility Cloud can ingest as a turn_event for compliance reporting.

Notes

Posture inference uses a torso-angle heuristic on the shoulder-hip line:
  • Standing: torso line near vertical (shoulder above hip)
  • Sitting: torso vertical but knee + hip nearly collinear
  • Lying: torso line near horizontal — flag for in-bed monitoring
For clinical-grade fall detection on top of these keypoints, the upgrade path is PySKL + PoseC3D trained on NTU-RGB+D.

Privacy & deployment

This card uses @mediapipe/tasks-vision via CDN. WASM + the pose-landmarker model load on first start. After that, every frame is processed in-browser via WebAssembly — the camera stream never leaves the device, and the rendered video element is the only place RGB bytes exist. The skeleton landmarks (33 points × frame) are the only data kept in memory, which means we can safely persist or stream those to BMM/UMM for trend reporting without exfiltrating identifying video.

Production deployment: drop a Jetson Orin Nano + Intel RealSense D435i overhead in patient rooms. RTMPose ONNX (Apache-2.0) at 30–60 FPS replaces MediaPipe for headless inference; keypoints stream over WebSocket back into the b10-playground for unified visualization.

How to add a new toolkit (engineering recipe)

  1. Drop a new file in sidecar/algorithms/ (e.g. gait_mobgap.py) that subclasses BaseAlgorithm from algorithms/base.py.
  2. Implement metadata (name, license, upstream URL, expected sample rate) and process(samples) -> dict.
  3. Register it in algorithms/__init__.py.
  4. Restart the sidecar. The algorithm appears at /v1/analyze/<name> and /v1/stream/<name>, plus in the dropdown above. OpenAPI docs auto-update.

Auth

Default config password: minew123 (exactly 8 chars). Minew's SDK docs warn that wrong-length passwords can brick firmware — we only send the default.

6 advertising SLOTs

The B10 broadcasts up to 6 concurrent advertisement channels. Each slot can be bound to a trigger (button tap, motion, always) and configured to emit one frame type.

Frame typesiBeacon · Eddystone UID · Eddystone URL · Eddystone TLM · Info · Acceleration · Empty
TriggersAlways · Button STAP · DTAP · TTAP · Push/Release · Motion
Interval100–5000 ms per slot
TX power−8 / −4 / 0 / +4 dBm
SLOT 5⚠️ Reserved. Required INFO packet for BeaconSET+ to connect.
